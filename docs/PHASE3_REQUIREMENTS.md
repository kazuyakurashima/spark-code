### 0.5 Claude Code での実装運用方針

本書は、Claude Code に一括実装させるための雑な依頼書ではない。

Claude Code には、以下の順序で作業させる。

1. 本書を読む
2. まず実装せず、Phase 3.1 のみを対象に TODO リストを作る
3. TODO ごとに、本書のどの要件に対応するかを明記する
4. 1タスクずつ実装する
5. 各タスク完了後に、要件定義書通りになっているか自己チェックする
6. ズレがあれば、次のタスクに進む前に修正する
7. Phase 3.1 完了後に Codex Review を実行する

重要:

- Phase 3.1 完了前に、Phase 3.2 以降の実装に着手してはならない
- ただし、Phase 3.2 以降のために必要な TODO や TODO コメントを残すことは許可する
- 実装の軽さを理由に、スモールステップ・Sparkコーチ・3周構造・成長実感・課金導線を削ってはならない
- 判断に迷った場合は、本書 §19「最重要メッセージ」を優先する

## 1. プロダクト定義

### 1.1 プロダクト名

SparkCode

### 1.2 プロダクトの一文定義

SparkCode は、完全初心者が Sparkコーチと一緒に、自分の自己紹介カードを 3 周かけて育てる学習サービスである。

### 1.3 プロダクトが提供する体験

ユーザーが以下の状態になることを目指す:

- 自分にもコードを書けると思える
- HTML/CSS/JavaScript の全体像を体験として理解できる
- 1 レッスンごとに小さな成功を感じられる
- 自分のカードが少しずつ育つ感覚を持てる
- つまずいても Sparkコーチに聞けば進めると感じられる
- もっと学びたい、このまま続けたい、課金してでも完成させたいと思える

### 1.4 SparkCode が提供しないもの(明示的な除外)

- HTML/CSS/JavaScript を網羅的に教える教材としての価値
- フレームワーク(React, Vue 等)の習得
- バックエンド開発の習得
- 就職や副業のためのスキル証明書

SparkCode は「**作る楽しさを通じて全体像を体験するサービス**」であり、技術習得を主目的とする教材ではない。

## 2. 対象ユーザー(仮ペルソナ)

### 2.1 主対象

- プログラミング完全初心者
- 大学生〜若手社会人
- HTML/CSS/JavaScript をほとんど触ったことがない人
- 自分で何かを作ってみたいが、何から始めればよいかわからない人

### 2.2 想定する心理状態

- コードに対して不安がある
- エラーが出ると止まりやすい
- 技術名よりも「何ができるようになるか」を知りたい
- 一人だと挫折しやすい
- 褒められたり、次の一歩を示されたりすると継続しやすい

### 2.3 検証フェーズの想定

Phase 3.1 完了時点で、以下の検証を行う想定:

- 大学生 3 名(運営者の知人、教え子)
- 60 代退職後 2 名(UI 視点・初心者視点の補助検証)
- 必要に応じて 20 代社会人の知人 1-2 名

ペルソナは固定せず、検証データから修正していく。

## 3. 学習設計の原則

すべてのレッスンは、以下の 5 原則に従う。

### 3.1 1レッスン1新概念

1 つのレッスンで、新しく教える主要概念は 1 つにする。

| 例 | 良い | 悪い |
|---|---|---|
| HTML 基礎 | 自己紹介文を書く | 自己紹介文と画像とリンクを同時に追加する |

### 3.2 1レッスン1小成功

各レッスンの最後に、ユーザーが「できた」と感じられる変化を必ず作る。

例:

- 名前が画面に出た
- 色が変わった
- 角が丸くなった
- 入力した名前がカードに反映された

### 3.3 1レッスン1つの見た目変化

毎レッスンで、プレビュー上に目に見える変化を出す。

ユーザーは技術そのものではなく、自分の作品が変化することに楽しさを感じる。

### 3.4 1レッスン1つの Sparkコーチフィードバック

各レッスンの最後に、Sparkコーチが以下の 3 点を返す。**これを「3 点セット」と呼ぶ。**

1. 今日できるようになったこと
2. あなたのカードがどう進化したか
3. 次のレッスンで何が楽しみか

3 点セットの実装詳細は §9.6 参照。

### 3.5 技術名よりも変化を先に出す

初心者向けの説明では、技術名を先に出さない。
ただし、変化は「お膳立て CSS の派手さ」ではなく、**学習者が書いたコードで画面が変わる事実**で起こす。Phase 3.1 の Lesson 1〜6 では、初期装飾を最小限にして、HTML / CSS / JavaScript それぞれの役割が見えるようにする。

| 場面 | 悪い説明 | 良い説明 |
|---|---|---|
| Lesson 9 | 今日は `border-radius` を学びます | 今日はカードの角を丸くして、やわらかい印象にします |
| Lesson 12 | 今日は `addEventListener` を学びます | 今日は入力した名前が、カードに反映されるようにします |
| Lesson 11 | 今日は `transition` を学びます | 今日は触ると少し動くカードを作ります |

## 4. 全体構成

### 4.1 3周構成の哲学

SparkCode は 3 周構成とする。**これは「HTML → CSS → JavaScript」の 3 段階ではない。**

**同じ自己紹介カードを、3 回育てる構成にする。**

| 周回 | 目的 | ユーザー体験 |
|---|---|---|
| 1 周目 | Web 制作の全体像を体験する | HTML/CSS/JS/Sparkコーチを浅く体験する |
| 2 周目 | カードを作品らしく整える | 見た目が大きく変わり、作品感が出る |
| 3 周目 | カードを動くアプリにする | 入力・反映・カスタマイズができるようになる |

### 4.2 3周設計の構造的原則

| 設計 | 内容 |
|---|---|
| 悪い設計(3 段階) | 1 周目で HTML だけ、2 周目で CSS だけ、3 周目で JS だけ |
| 良い設計(真の 3 周) | 1 周目で扱った要素を、2 周目で装飾し、3 周目で動的にする |

**1 周目で登場する要素は、2 周目・3 周目で必ず再訪する。**

これにより、学習者は同じ要素に 3 回触れ、知識が定着する。詳細は §5.2 のカード進化マトリクス参照。

### 5.2 カード進化マトリクス

**1 周目で登場した各要素が、2 周目・3 周目でどう進化するかの一覧。**

このマトリクスが「3 段階」ではなく「3 周」であることを担保する。

| 要素 | 1 周目で登場 | 2 周目で進化 | 3 周目で進化 |
|---|---|---|---|
| **名前** `<h1>` | L1: 表示する | L8: 周りに余白 / L9: フォントサイズ調整可 | L12: 入力欄から反映 |
| **自己紹介文** `<p>` | L2: 表示する | L8: 余白で読みやすく | L13: 入力欄から反映 |
| **好きなもの** `<ul><li>` | L3: 表示する | L9: 角丸の背景タグ風に装飾 | (発展課題、Phase 3.4 で項目追加機能を検討) |
| **色** | L4: 文字色を 1 つ変える | L10: テーマ色から選択 / Tailwind 表記 | L14: ボタンで動的に変える |
| **動き** | L5: 1 行 JS で文字変更を体験 | L11: hover で触ると反応 | L12-14: 自分で書く JS |
| **カード構造** | (未登場) | L7: `<div>` でまとめる | (構造維持、要素のみ動的化) |
| **影と装飾** | (未登場) | L9: shadow / radius | L11 で hover 時に影が深まる |

**この表を読むと、各要素が 3 回登場することがわかる。これが定着を生む。**

## 6. 1周目詳細(Lesson 1〜6、無料)

### Lesson 1:名前が画面に出る

- **目的**: HTML を書いたら画面に表示される、という最初の成功体験を作る
- **主な技術**: `<h1>`
- **ユーザーが書くコード例**:
  ```html
  <h1>かず</h1>
  ```
- **判定**: `<h1>` タグが存在し、中身が空でない
- **ユーザー体験**: 自分の名前が大きく表示される
- **Sparkコーチの 3 点セット例**:
  > 今日できるようになったこと: HTML で名前を画面に表示できました。
  > あなたのカードの進化: 何もなかった画面に、あなたの名前が生まれました。
  > 次の楽しみ: 次は自己紹介文を追加して、もっとあなたらしいページにします。

### Lesson 2:自己紹介文が出る

- **目的**: 名前だけでなく、文章を追加してページらしくする
- **主な技術**: `<p>`
- **ユーザーが書くコード例**:
  ```html
  <h1>かず</h1>
  <p>水戸の塾で先生をしています</p>
  ```
- **判定**: `<p>` タグが存在し、中身が空でない
- **ユーザー体験**: 自己紹介文が表示される、自分のページ感が出る

### Lesson 3:好きなものリストが出る

- **目的**: 自分らしさを表現できる要素を追加する
- **主な技術**: `<ul>`, `<li>`
- **ユーザーが書くコード例**:
  ```html
  <h1>かず</h1>
  <p>水戸の塾で先生をしています</p>
  <ul>
    <li>司馬遼太郎</li>
    <li>歴史</li>
    <li>ジャズ</li>
  </ul>
  ```
- **判定**: `<ul>` 内に `<li>` が 2 つ以上存在
- **ユーザー体験**: 好きなものを並べて、自分らしさが出る

### Lesson 4:色が変わる

- **目的**: CSS によって見た目が変わることを体験する
- **主な技術**: `color`
- **重要**: ここで初めて `<style>` タグが登場する。CSS を書く場所として導入する
- **ユーザーが書くコード例**:
  ```html
  <style>
    h1 { color: pink; }
  </style>
  <h1>かず</h1>
  ...
  ```
- **判定**: `<style>` 内に `color` プロパティが存在し、初期値(black)以外に変更されている
- **ユーザー体験**: 文字色が変わる、CSS は見た目を変えるものだと体験で理解する

### Lesson 5:1 行 JS で文字が変わる

- **目的**: JavaScript によって画面の内容が変わることを、最小負荷で体験する
- **主な技術**: `textContent`
- **重要**: `addEventListener`、`querySelector`、`input.value` は **学習者に書かせない**。
- **実装方針**: `document.querySelector` はシステム側のお膳立てコードとして非表示、または説明対象外にする。
- **学習者が書くコードは必ず 1 行だけ**に固定する。
- **ユーザーが書くコード例**:
  ```js
  name.textContent = "こんにちは!";
  ```

  お膳立てコード例:
  ```js
  // この行はシステム側で用意し、学習者には書かせない
  const name = document.querySelector("#name");

  // 学習者が書くのはこの1行だけ
  name.textContent = "こんにちは!";
  ```

- **判定**: `textContent` を使った代入式が存在する
- **ユーザー体験**: 自分が 1 行コードを書いたことで、画面の文字が変わる
- **Sparkコーチからの予告**: 「これは JavaScript の世界です。3 周目で自分でもっと書けるようになります」

### Lesson 6:1 周目クリアと総合振り返り

- **目的**: 1 周目の達成感を作り、課金導線につなげる
- **内容**:
  - 1 周目で扱った 5 つの要素(名前・自己紹介・リスト・色・動き)の振り返り
  - HTML / CSS / JavaScript / Sparkコーチの全体像確認
  - Sparkコーチからの総合振り返り
  - 「未来のカード」プレビュー(今のカードのサンプルと、2 周目・3 周目で実現する装飾・動きのサンプルを見せる)
  - 課金誘導: 「SparkPlus でカードを育てる」ボタン
- **重要**: ここでは新しいコードは書かせない。「祭り」のレッスン
- **Sparkコーチの総合振り返り例**:
  > ここまでで、あなたは Web 制作の全体像を一周できました。
  > HTML で中身を作り、CSS で見た目を変え、JavaScript で画面を変化させました。
  > わからないところは私(Sparkコーチ)に聞きながら、ここまで進んでこられましたね。
  > ここから先は、このカードをもっと作品らしく育てていきます。
  > 角丸、影、テーマ色、入力で変わる仕組みまで。
  > 一緒に進みましょう。

## 7. 2周目詳細(Lesson 7〜11、有料)

### Lesson 7:カード型にまとめる

- **目的**: 要素をひとまとまりにし、作品の器を作る
- **主な技術**: `<div>`
- **ユーザーが書くコード例**:
  ```html
  <div class="card">
    <h1>かず</h1>
    <p>水戸の塾で先生をしています</p>
    <ul>...</ul>
  </div>
  ```
- **お膳立てされたCSS**: `.card` には事前に `width: 320px; margin: auto; background: white;` 程度を入れておく
- **判定**: `<div>` で全要素が囲まれている
- **ユーザー体験**: バラバラの要素がカードとしてまとまる

### Lesson 8:余白を整える

- **目的**: 見やすさを改善する
- **主な技術**: `padding`, `margin`
- **進化対象**: 名前(L1)、自己紹介(L2)、リスト(L3)の周りに余白を作る
- **ユーザーが書くコード例**:
  ```css
  .card { padding: 24px; }
  h1 { margin-bottom: 8px; }
  ```
- **判定**: `padding` または `margin` が `.card` または子要素に適用されている
- **ユーザー体験**: 窮屈だったカードが見やすくなる

### Lesson 9:角丸と影をつける

- **目的**: カードらしさを一気に出す
- **主な技術**: `border-radius`, `box-shadow`
- **進化対象**: カード全体、好きなものリスト(L3)の各項目に背景色 + 角丸で「タグ風」にする
- **ユーザーが書くコード例**:
  ```css
  .card {
    border-radius: 16px;
    box-shadow: 0 4px 20px rgba(0,0,0,0.1);
  }
  li {
    background: #fce7f3;
    padding: 4px 12px;
    border-radius: 999px;
    display: inline-block;
  }
  ```
- **判定**: `border-radius` と `box-shadow` の両方が適用されている
- **ユーザー体験**: カードがプロっぽくなる、見た目の変化が大きい

### Lesson 10:背景・テーマ色とTailwind翻訳

- **目的**: CSS で書いたことを Tailwind で表現できることを理解する
- **主な技術**: `background`, Tailwind utility classes
- **重要**: Tailwind を「**CSS の翻訳**」として導入する。突然出さない
- **CSS と Tailwind の対応表(レッスン内で表示)**:

  | CSS | Tailwind |
  |---|---|
  | `color: white;` | `text-white` |
  | `background-color: #ec4899;` | `bg-pink-500` |
  | `padding: 16px;` | `p-4` |
  | `border-radius: 12px;` | `rounded-xl` |
  | `box-shadow: 0 4px 20px rgba(0,0,0,0.1);` | `shadow-lg` |

- **ユーザー体験**:
  1. まず Lesson 9 までの CSS をそのまま見せる
  2. Sparkコーチが「**プロは同じことを Tailwind で書きます。短くて読みやすいです**」と紹介
  3. 同じカードを Tailwind で書き直す
  4. テーマ色を `bg-pink-500` から `bg-blue-500` に変えるだけで雰囲気が変わることを体験
- **判定**: Tailwind クラスが少なくとも 3 つ適用されている

### Lesson 11:hoverで触りたくする

- **目的**: 触ると反応する楽しさを作る
- **主な技術**: `hover:`, `transition`
- **進化対象**: カード全体、リストの各タグ
- **ユーザーが書くコード例**:
  ```html
  <div class="card hover:shadow-2xl transition">
    ...
    <li class="hover:bg-pink-300 transition">司馬遼太郎</li>
  </div>
  ```
- **判定**: `hover:` クラスが適用され、`transition` も含まれている
- **ユーザー体験**: マウスを乗せると影が深くなる、タグの色が変わる、プロっぽい

## 8. 3周目詳細(Lesson 12〜16、有料)

### Lesson 12:入力した名前を反映する

- **目的**: 入力欄の文字をカードに反映する
- **主な技術**: `<input>`, `value`, `addEventListener`, `textContent`
- **進化対象**: 名前(L1, L8)を入力で動的に変更
- **ユーザーが書くコード例**:
  ```html
  <input id="name-input" placeholder="名前を入力" />
  <h1 id="name">かず</h1>
  <script>
    const input = document.querySelector("#name-input");
    const name = document.querySelector("#name");
    input.addEventListener("input", () => {
      name.textContent = input.value;
    });
  </script>
  ```
- **重要**: ここで初めて `addEventListener` と `input.value` を導入する。Lesson 5 の予告が回収される
- **判定**: `addEventListener` が `<input>` に対して設定され、`textContent` で書き換えが行われている
- **ユーザー体験**: 自分で入力した名前にカードが変わる、Web ページから Web アプリに近づく

### Lesson 13:自己紹介文も反映する

- **目的**: 名前だけでなく、自己紹介文も変更できるようにする
- **主な技術**: 既存処理の応用、`<textarea>` の導入
- **進化対象**: 自己紹介文(L2, L8)を入力で動的に変更
- **ユーザー体験**: 自分のカードを編集できる感覚が強まる
- **学習ポイント**: 「同じパターンを繰り返す」体験。L12 で覚えたコードを応用するだけ

### Lesson 14:色を選んで反映する

- **目的**: 見た目を自分でカスタマイズできるようにする
- **主な技術**: `event`, `classList`, テーマ選択
- **進化対象**: テーマ色(L10)をボタンで動的に変える
- **ユーザーが書くコード例**:
  ```html
  <button id="pink">ピンク</button>
  <button id="blue">青</button>
  <div id="card" class="card bg-pink-500">...</div>
  <script>
    document.querySelector("#blue").addEventListener("click", () => {
      const card = document.querySelector("#card");
      card.classList.remove("bg-pink-500");
      card.classList.add("bg-blue-500");
    });
  </script>
  ```
- **判定**: `classList` を使った操作が含まれている
- **ユーザー体験**: 好きな色を選べる、自分だけのカード感が強まる

### Lesson 15:Sparkコーチの仕上げレビュー

- **目的**: 完成前に、自分のコードやカードを Sparkコーチに見てもらう
- **重要**: このレッスンでは新しいコードは書かない。**Sparkコーチが学習者のカード全体を評価するレッスン**
- **Sparkコーチのレビュー観点**:
  - よくできている点(具体的に 2-3 つ)
  - コード上の改善点(あれば、なくても OK)
  - 見た目の改善案(任意)
  - これまでの 14 レッスンで成長した点
- **実装方針**: 学習者の最終コードと学習ログ(`learning_events`)を渡し、Sparkコーチが評価コメントを返す
- **ユーザー体験**: 自分の成長を実感できる、自分の作品に対するフィードバック

### Lesson 16:完成・共有・発表

- **目的**: 完成体験と共有体験を作る
- **内容**:
  - 完成画面を表示(BGM や紙吹雪エフェクトでも良い)
  - **共有用 URL を発行**(独自ドメインまたは sparkcode.app/cards/xxx 形式)
  - SNS 投稿文(コピー可能テキスト)を生成
  - Sparkコーチが最後のコメントを返す
- **Sparkコーチの最後のコメント例**:
  > 完成おめでとうございます。
  > あなたは、HTMLで中身を作り、CSSとTailwindで見た目を整え、JavaScriptで動きを加えました。
  > これはもう、ただの練習ではなく、あなたが自分で育てたWeb作品です。
  > URL を友達やSNSにシェアすれば、世界中の誰からでも見られます。

## 9. Sparkコーチ要件

### 9.1 Sparkコーチの定義

Sparkコーチは、単なるチャット欄ではない。

**学習者の不安を減らし、コードのつまずきを解消し、成長実感を言語化し、次の一歩を示す伴走 AI である。**

SparkCode の中心価値であり、Sparkコーチを「おまけ」として実装してはならない。

### 9.2 Sparkコーチの 4 役割

| 役割 | 内容 | 対応する常設ボタン | 対応する内部 API |
|---|---|---|---|
| **先生** | わからない概念をやさしく説明する | やさしく説明して | `question` |
| **コード診断** | 書いたコードのミスを見つける | どこが違う? | `judge`(改良版) |
| **応援者** | できたことを言語化して褒める | できたことを教えて | `praise` |
| **ナビゲーター** | 次に何をすればよいか示す | ヒントがほしい / もっと良くしたい | `hint` |

### 9.3 常設ボタン仕様

Sparkコーチパネルには、以下の 5 つのボタンを常設する。

| ボタン | 役割 | 動作 |
|---|---|---|
| ヒントがほしい | ナビゲーター | 次の 1 ステップだけを示す。完成形は見せない |
| どこが違う? | コード診断 | 学習者のコードと正解パターンを比較し、差分を 1 か所だけ指摘 |
| やさしく説明して | 先生 | 現在のレッスンの主要概念を、初心者向けに 3-5 行で説明 |
| できたことを教えて | 応援者 | 学習ログから、3 つの「できるようになったこと」を抽出 |
| もっと良くしたい | ナビゲーター(改善版) | 現在のコードに対して、次のレッスンの予告を 1 つ示す |

### 9.4 自由質問欄

常設ボタンに加えて、自由入力欄も用意する。

ただし、初心者は質問文を作ること自体に詰まるため、**常設ボタンを優先的に表示**し、自由入力欄はその下に配置する。

### 9.5 Sparkコーチの返答トーン

- やさしい
- 短すぎず、長すぎない(3-7 行が目安)
- 具体的(「がんばろう」だけは避ける)
- 何を直せばよいか明確
- できたことを必ず認める
- 技術名を使う場合は、必ず意味も添える(例: `<h1>` (見出しタグ))

#### 悪い返答例

> 間違っています。

#### 良い返答例

> 惜しいです。`<h1>` の閉じタグが抜けています。
> HTML では、始めたタグは最後に閉じるのが基本です。
> `<h1>かず</h1>` の形にすると、名前が正しく表示されます。
> あと一歩です。

### 9.6 各レッスン末尾の 3 点セット

各レッスン完了時(判定が正解になった瞬間)に、Sparkコーチが以下を自動で返す。

```
今日できるようになったこと: ○○○
あなたのカードが進化したこと: ○○○
次の楽しみ: ○○○
```

#### 実装方針

- 3 点セットの内容は、各レッスンごとにテンプレ文として用意(AI 不使用、判定後にテンプレを差し込む)
- ただし、学習者の苦労度合い(ヒント使用回数、リトライ回数)に応じて、文面を 2-3 種類用意して切り替える(例: 一発正解 / 数回トライ / かなり苦労)
- AI を使うのは「やさしく説明して」「どこが違う?」「もっと良くしたい」のみ

### 9.7 Sparkコーチプロンプト設計

各役割のプロンプト設計の方針:

#### 9.7.1 先生プロンプト(question / やさしく説明して)

```
あなたはSparkコーチです。完全初心者にやさしく教える先生です。
現在のレッスン: {lesson_title}
現在のレッスンの主要概念: {concept}
ユーザーが知りたいこと: {user_question or "現在のレッスンの主要概念をやさしく教えて"}

返答ルール:
- 3-5 行で説明する
- 技術名を使う場合は意味も添える(例: `<h1>` (見出しタグ))
- 例を 1 つ示す
- できたら今のコードに紐づけて説明する
```

#### 9.7.2 コード診断プロンプト(judge / どこが違う?)

```
あなたはSparkコーチです。
現在のレッスン: {lesson_title}
正解パターン: {expected_pattern}
ユーザーのコード: {user_code}

判定:
1. 正解なら、何が良かったかを 2 行で褒める
2. 不正解なら、差分を 1 か所だけ指摘し、修正例を 1 行示す
3. 複数の問題があっても、最重要なもの 1 か所だけ指摘する

返答ルール:
- 「間違っている」と言わず「惜しい」「あと一歩」を使う
- できている部分も認める
- 技術用語を使う場合は意味も添える
```

#### 9.7.3 応援者プロンプト(praise / できたことを教えて)

```
あなたはSparkコーチです。
学習者の最近の学習ログ:
{learning_events 直近 20 件}

このログから、以下を抽出して 3 つの「できるようになったこと」を返す:
- 完了したレッスン
- 苦労した後にクリアしたステップ
- 自分から質問して解決した内容

返答ルール:
- 3 つを箇条書き
- 「○○ができるようになりましたね」と過去形で伝える
- 最後に 1 行、励ましのコメント
```

#### 9.7.4 ナビゲータープロンプト(hint / ヒントがほしい)

```
あなたはSparkコーチです。
現在のレッスン: {lesson_title}
ユーザーの現在のコード: {user_code}
正解パターン: {expected_pattern}

ヒントを返す:
- 完成形は絶対に示さない
- 次の 1 ステップだけを示す
- ユーザーが既に書けている部分は触れない

返答ルール:
- 「次に〜してみましょう」の形で 1 文
- 必要なら例を 1 行(ただし全部書かない)
```

#### 9.7.5 もっと良くしたいプロンプト(hint 拡張)

```
あなたはSparkコーチです。
現在のレッスン: {lesson_title}
ユーザーが今書いたコード: {user_code}
次のレッスン: {next_lesson_title}

返答:
- 現在のコードでよくできている点を 1 行
- 次のレッスンで実現できる「もっと良くする 1 つの提案」
- ただし具体的なコードは示さない、予告だけ
```

### 9.8 学習者の状態に応じた対応

Sparkコーチは、学習者の状態(`learning_events` のログ)を見て、対応を変える:

| 状態 | 対応 |
|---|---|
| 同じレッスンで 3 回以上失敗 | 「どこが違う?」を促す通知を自動で表示 |
| 5 分以上コード変更がない | 「ヒントがほしい?」を控えめに表示 |
| ヒント使用なしで連続 3 レッスン正解 | 「すごい、調子いいですね」と褒める |
| 1 周目クリア後 | Lesson 6 の総合振り返りを実行 |

## 12. localStorage と保存設計

### 12.1 基本方針

**Phase 3.1 では localStorage を本編必須機能として扱わない。**

理由:

- 初心者向けの初期学習では、保存 / 読み込みの概念は本筋ではない
- key/value、初期表示などの概念が増えると認知負荷が上がる
- 自己紹介カードの初期価値は「作る・変わる・動く・コーチが支える」にある

### 12.2 ユーザーのコード保存

学習者のコードは **Supabase に保存する**。理由:

- ユーザーが別端末で続きをやれる
- 公開 URL の発行が可能になる
- 学習ログとセットで保存できる
- 将来の認証システム導入時に自然

実装の詳細:

- ユーザー識別は session_id(localStorage に保存される uuid)で行う(認証導入前)
- コード変更時、デバウンス(2 秒)で Supabase に保存
- `user_code` テーブルを新設(session_id, lesson_id, code, updated_at)

### 12.3 公開 URL

- Lesson 16 完成時に発行
- URL 形式: `https://spark-code-mu.vercel.app/cards/{share_id}`
- share_id はランダムな短い文字列(8 文字程度)
- Lesson 6 の段階での「公開 URL」は **Phase 3.4 までは見送り**(プレビュー画面の表示のみ)

### 12.4 認証(将来)

- Phase 3.2 で SparkPlus 課金と同時に、認証システムを導入
- 無料ユーザーは session_id ベース、有料ユーザーはアカウントベース
- 既存の session_id データは、認証時にアカウントに紐づける

## 13. Claude Code 実装運用ルール

### 13.1 実装前に必ず行うこと

Claude Code は、実装を開始する前に必ず以下を行う。

1. `docs/PHASE3_REQUIREMENTS.md` を読む
2. `docs/PROJECT_OVERVIEW.md` を読む
3. 現在の実装状態を確認する
4. Phase 3.1 のみを対象に TODO リストを作成する
5. 各 TODO に対応する要件番号を紐づける
6. 影響範囲と変更予定ファイルを明記する
7. 実装前に TODO リストを提示する

この段階では、まだ実装しない。

### 13.2 TODO リストの作成ルール

Claude Code は `TODO_PHASE3.md` を作成する。

TODO は以下の形式にする。

```md
# TODO_PHASE3.md

## Phase 3.1 実装TODO

### T1: 現在地表示バーを実装する
- 対応要件: §10.2
- 目的: 初心者が今どこにいるか迷わないようにする
- 変更予定ファイル:
  - components/...
  - app/...
- 完了条件:
  - 現在の周回が表示される
  - Lesson番号が表示される
  - HTML/CSS/JS/Sparkコーチの凡例が表示される
- 要件適合チェック:
  - [ ] §10.2 の表示要素を満たしている
  - [ ] 初心者に現在地が伝わる
  - [ ] 既存UIを壊していない
```

#### TODO のまとめて作成、まとめてレビュー運用

実運用上、Claude Code に **TODO を 1 つずつ提示してもらうのは効率が悪い**。
以下の運用を推奨する:

- **TODO 作成**: Phase 3.1 全体(15〜25 個程度想定)を**一気に作成**して提示する
- **かずのレビュー**: 全 TODO を**5 タスクずつグループ化**して確認する(全部個別レビューは時間を食う)
- **承認後の実装**: 承認された順に Claude Code が実装する

このルールにより、TODO 作成段階での無駄なやり取りを減らし、レビュー時間も圧縮できる。

### 13.3 タスク分割の粒度

1つの TODO は、原則として 30〜90分程度で完了できる大きさにする。

#### 時間目安の内訳

「30〜90 分」は、以下の合計目安である:

| 内訳 | 時間 |
|---|---|
| Claude Code 単独の実装時間(YOLO モード) | 10〜30 分 |
| かずによるレビュー時間 | 10〜30 分 |
| 修正・調整の時間 | 10〜30 分 |
| **合計目安** | **30〜90 分** |

Claude Code は実装速度が速いが、**レビューと調整に時間がかかる**前提で粒度を切る。
1 タスクが 90 分を超えそうなら、サブタスクに分割する。

悪い分割:

- Phase 3.1 を全部実装する

### 16.1 レッスンを重くしない

## 17. ユーザー検証との連動

### 17.1 検証タイミング

Phase 3.1 完了直後に、ユーザー検証を実施する想定。

#### 検証対象

- 大学生 3 名(運営者の知人、教え子)
- 60 代退職後 2 名(UI 視点・初心者視点の補助)
- 必要に応じて 20 代社会人の知人 1-2 名

#### 検証する 5 つの問い

1. 最後まで完成できたか(Lesson 6 まで)
2. 完成したものを誰かに見せたいと思ったか
3. どこで一番つまずいたか
4. 続き(Lesson 7 以降)があったらやりたいか
5. 月 498 円・980 円なら払うか

#### 検証の合格基準

Phase 3.1 の目的に照らし、以下を暫定合格ラインとする。

検証対象人数の前提:**大学生 3 名 + 60 代 2 名 = 計 5 名**

| 検証項目 | 合格ライン | 判断 |
|---|---|---|
| Lesson 1〜6 完走率 | 80%(5人中4人以上) | 未達ならレッスン粒度を再調整 |
| 同じ箇所でのつまずき | 60%(5人中3人以上が同一箇所で停止)で要修正 | そのレッスンの説明・判定・Sparkコーチを修正 |
| Sparkコーチ利用率 | 80%(5人中4人以上が1回以上利用) | 未達ならボタン配置・文言を改善 |
| 1周目後の全体像理解 | 80%(5人中4人が HTML/CSS/JS の役割を説明できる) | 未達なら Lesson 6 を修正 |
| 続きをやりたい意向 | 60%(5人中3人以上) | 未達なら未来プレビューと課金前体験を修正 |
| 498円課金意向 | 40%(5人中2人以上なら有望) | 0〜1人なら課金訴求を見直す |

合格ラインは**比率でも判断可能**にしているため、検証対象人数が変わっても適用できる(例: 検証 8 人中 6 人完走で 75%、ラインの 80% 未達と判定)。

この合格基準は、Phase 3.2 以降の優先順位判断に使う。

### 17.2 検証結果を Phase 3.2 以降に反映

検証で得られた知見に基づき、Phase 3.2 以降の優先順位を調整する。

例:

- 大学生が「Sparkコーチの常設ボタンは 5 個多すぎる」と感じれば → 3 個に絞る
- 60 代が「現在地表示バーが小さすぎる」と感じれば → 文字サイズを上げる
- 「月 498 円なら払う」が 3 人中 0 人 → 課金訴求コピーを根本的に見直す

### 17.3 検証中の運用

- ログイン不要(session_id で動作)
- Sparkコーチが録画する(将来の改善のため、ただしプライバシー配慮、本人同意の上)
- 検証セッション後 24 時間以内に学習ログを分析

## 20. 最重要メッセージ

SparkCode は、HTML/CSS/JavaScript を順番に教える教材ではない。

**完全初心者が、Sparkコーチと一緒に、自分の自己紹介カードを 3 周かけて育てる学習サービスである。**

- 一人ではない安心感
- いつでも聞ける伴走感
- 自分のコードを見てもらえる安心感
- 自分のカードが育っていく楽しさ
- 最後まで完成できそうという期待感

この体験を中心に実装すること。
